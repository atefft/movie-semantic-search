MakefilePipelineTarget

#113 — Update documentation for the two-phase pipeline workflow

Spec for the pipeline Makefile target (Makefile) — orchestrates the two-phase data loading workflow with per-phase error reporting.

Overview

The pipeline Make target is the single-command entry point for initialising the full system from scratch. It sequences three operations in strict order: load-model (export the ONNX model into the Triton model repository), services-up (docker compose up -d to start Triton, Qdrant, and the API), and load-data (embed all movie summaries via Triton and ingest them into Qdrant with TMDB enrichment). If any phase exits non-zero the target prints a clearly labelled message identifying the failed phase and aborts, preventing silent partial runs that are hard to diagnose.

The old pipeline target started Triton before exporting model.onnx. Triton would fail to load the model because the file did not yet exist. The new target fixes this ordering bug by running load-model first.

Implementation

# Makefile pipeline: docker compose run --rm load-model \ || (echo "[pipeline] load-model phase failed — aborting" && exit 1) docker compose up -d \ || (echo "[pipeline] services-up phase failed — aborting" && exit 1) docker compose run --rm load-data \ || (echo "[pipeline] load-data phase failed — aborting" && exit 1)

Phase Sequence

#PhaseCommandDescriptionStatus
1 load-model docker compose run --rm load-model Exports all-MiniLM-L6-v2 to ONNX and places files in the Triton model repository. Must complete before Triton starts. phase 1
2 services-up docker compose up -d Starts triton, qdrant, and api containers in detached mode. Health checks must pass before load-data can communicate with Triton and Qdrant. phase 2
3 load-data docker compose run --rm load-data Downloads the CMU corpus, embeds all plot summaries via Triton (gRPC), ingests vectors into Qdrant, and enriches with TMDB poster URLs. phase 3

Error Behaviour

Phase that failsError message printedExit code
load-model [pipeline] load-model phase failed — aborting 1
services-up [pipeline] services-up phase failed — aborting 1
load-data [pipeline] load-data phase failed — aborting 1
Subsequent phases do not run after a failure. The partial environment is left in place so the operator can inspect logs with docker compose logs before retrying.

Acceptance Criteria

  • make pipeline runs docker compose run --rm load-model first
  • make pipeline then runs docker compose up -d
  • make pipeline then runs docker compose run --rm load-data
  • If load-model exits non-zero, [pipeline] load-model phase failed — aborting is printed and make exits 1 without running later phases
  • If docker compose up -d exits non-zero, [pipeline] services-up phase failed — aborting is printed and make exits 1
  • If load-data exits non-zero, [pipeline] load-data phase failed — aborting is printed and make exits 1
