PipelineDockerfile

#111 Create pipeline/Dockerfile for containerized script execution

pipeline/Dockerfile — shared base image for all pipeline Compose services

Builds a python:3.11-slim-based image. Installs libgomp1 unconditionally via apt-get (required by torch CPU wheels on slim images), then installs all Python dependencies from pipeline/requirements.txt, and copies the five pipeline scripts into /app. Sets WORKDIR /app so that data/ and model-repository/ can be bind-mounted by Compose services. No ENTRYPOINT or CMD is defined — each Compose service specifies its own command.

Build context — no cd required Run docker build pipeline/ from the repo root — the argument is the build context path, not a directory to enter. In Compose, use build: { context: pipeline } and Compose resolves it automatically from the repo root. The only thing to avoid is docker build -f pipeline/Dockerfile ., which passes the repo root as context and fails at COPY requirements.txt ..

Dockerfile

pipeline/Dockerfile
FROM python:3.11-slim WORKDIR /app # libgomp1 required by torch CPU wheels on slim images RUN apt-get update \ && apt-get install -y --no-install-recommends libgomp1 \ && rm -rf /var/lib/apt/lists/* # Install Python deps; requirements copied first for layer-cache efficiency COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # Copy pipeline scripts; tests are excluded COPY *.py .

Normal usage is docker build pipeline/ from repo root or Compose build: { context: pipeline } — no cd required. Run with . as context deliberately to confirm the failure
2 docker run <image> with no command Container exits immediately No CMD/ENTRYPOINT is by design — Compose must supply command N/A — by design
3 Compose mounts ./data:/app/data, ./model-repository:/app/model-repository Scripts resolve pathlib.Path("data/raw") to /app/data/raw WORKDIR /app makes all relative script paths correct Standard Compose volume bind-mount
4 python -c "import torch" inside built image Exits 0, no ImportError libgomp1 is pre-installed unconditionally — no runtime library surprise docker run --rm pipeline-test python -c "import torch" exits 0
libgomp1 is a required system dependency python:3.11-slim strips libgomp1 (OpenMP runtime), which torch CPU wheels need at import time. The Dockerfile installs it unconditionally before pip install so the image is fully self-contained and predictable.

Unit Test Checklist

Dockerfile "unit tests" are build and smoke checks run with docker build and docker run:

  • docker build -t pipeline-test pipeline/ exits 0 (build succeeds)
  • docker run --rm pipeline-test python -c "import transformers, torch, onnxruntime, tritonclient.grpc, tqdm, qdrant_client, requests" exits 0 (all deps importable)
  • docker run --rm pipeline-test ls 01_download_corpus.py 02_export_model.py 03_embed_corpus.py 04_ingest_qdrant.py 05_enrich_tmdb.py exits 0 (all five scripts present in /app)
  • docker inspect pipeline-test --format '{{.Config.WorkingDir}}' returns /app
  • docker inspect pipeline-test --format '{{.Config.Entrypoint}}' returns [] (no entrypoint set)
  • docker inspect pipeline-test --format '{{.Config.Cmd}}' returns [] (no default command set)
  • Edge case #1: docker build -f pipeline/Dockerfile . from repo root fails with COPY error (confirming build context must be pipeline/)
